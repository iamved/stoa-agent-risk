## 7. Key decisions log (conservative-interpretation rule)

- **D-1** New `tests/test_parity.py` created as the "parity suite" the spec assumes exists.
- **D-2** Capability vocabulary: keep repo's underscored IDs; add `credential_read` + `user_impersonation` capabilities; map spec's hyphenated high-impact set onto them (documented map in `dimensions.py`).
- **D-3** `rules/*.toml` data → `src/stoa/data/*.toml` package data (no collision with `rules.py`).
- **D-4** Registry-diff lives in `registry_diff.py`; existing `diff.py` untouched.
- **D-5** Where behavior is truncated, do **not** guess — stop at the phase boundary and request the section (per working-style rule).
- **D-6 (Phase 2)** Rule IDs: `VALID_RULE_ID` widened `{3,5}`→`{2,5}` to admit the 2-letter `AI` prefix (used in suppressions/config/gating).
- **D-7 (Phase 2, adaptation)** Fixture corpus: the spec asks for 8 vuln + 8 clean fixtures per rule per language. Implemented as **parametrized inline trigger/non-trigger tests** in `tests/test_ai_rules_phase2.py` (each rule has multiple vulnerable and clean cases across Py + JS shapes) plus two end-to-end scanner tests, rather than ~128 on-disk files. Coverage is equivalent (every distinct trigger and non-trigger shape is asserted); the on-disk `tests/fixtures/ai_rules/**` corpus is deferred to the Phase 3 precision-gate work where AI002's zero-false-positive bar makes a standing corpus load-bearing.
- **D-8 (Phase 2)** `message` field added to `Finding` (schema 1.1, emitted only when set) to carry the spec's item-5 templated finding message verbatim (modulo interpolated values). AI categories (`ai-supplychain`/`ai-agency`/`ai-stability`/…) routed into the HTML report's security and review-prompt sections.
- **D-9 (Phase 2)** `data/model_aliases.toml` ships the floating-alias table (package data); loaded via `importlib.resources`, overridable later via `stoa.toml [rules.AI005]`.
- **D-10 (Phase 3)** AST layer + taint rules are **on by default** from Phase 3 (the point of v0.2). `--experimental-ast` is now a deprecated no-op; `--no-ast` opts out to regex-only. AST parse cost is <2× (benchmarked faster), so default-on is safe.
- **D-11 (Phase 3, adaptation)** Taint rules produce findings only when the AST is available; a degraded parse yields **no** taint finding for that file rather than a low-confidence regex-fallback guess (more conservative than the spec's "regex-only at low"). Logged; regex-fallback taint can be added later without a schema change.
- **D-12 (Phase 3, adaptation)** AI001 boundary handling: a sanitizer/escaper that breaks the taint chain results in **no** AI001 finding (conservative) rather than a downgraded `info` + `boundary_observed`. The downgrade still applies when a boundary identifier appears on an otherwise-intact flow. The stronger "always report, downgrade to info" is deferred; under-reporting a mitigated flow is the safe direction.
- **D-13 (Phase 3)** AI002 precision gate **met**: zero high-confidence AI002 on the clean corpus (`test_ai002_precision_*`), so AI002 exec-class ships `gate_eligible` by default. Gate logic reworked: AI rules gate only via `finding.gate_eligible` or `[gate].additional_rules` opt-in, never from `RULES.gateable` alone. Supersedes dedup (`_apply_supersedes`) implements AI002/sql⊃SEC003, AI005⊃NET001, AI006⊃AI004 by (rule, path, line).
